# 异常交易检测方法论

> 总结在实践中使用的异常交易检测方法，包括统计模型、机器学习方案和实际案例。

## 一、检测框架

异常交易检测 = **特征工程** + **检测模型** + **处置策略**

### 流程

```
原始交易流 → 特征提取 → 实时打分 → 规则引擎 → 处置动作
                ↓
           离线模型训练（定期更新）
                ↓
           回溯验证 → 调参 → 上线
```

## 二、特征工程

### 2.1 交易层特征

| 特征类别 | 具体特征 | 来源 |
|---------|---------|------|
| 量价特征 | 单笔金额、累计金额、买卖方向 | 订单簿 |
| 频率特征 | 1min/5min/1h 交易次数 | 滑动窗口计数 |
| 价格特征 | 成交价偏离度、买卖价差 | 行情数据 |
| 时间特征 | 交易间隔、活跃时段 | 时间戳 |

### 2.2 账户层特征

| 特征类别 | 具体特征 | 
|---------|---------|
| 行为特征 | 登录频率、设备切换次数、IP 变更频率 |
| 资产特征 | 余额变化率、充提频率、杠杆使用率 |
| 社交特征 | 关联账户数量、对手方集中度 |
| 历史特征 | 历史违规次数、近30天告警次数 |

### 2.3 关联特征

- **设备指纹聚类**：相同设备登录的账户群
- **IP 地址段聚合**：同 IP/C 段的账户关联
- **资金链路**：充提地址关联图
- **交易对手网络**：频繁交易对手对

## 三、检测模型

### 3.1 规则模型（Rule-based）

适用场景：已知攻击模式、合规要求、快速响应

**示例规则：**

```
Rule: 大额对敲检测
  IF 相同IP段 AND 买卖对手互为对方
     AND 成交价偏离 < 0.1%
     AND 1h 内成交次数 > 10
  THEN 告警 + 人工审核

Rule: 幌骗检测
  IF 挂单撤单率 > 95%
     AND 撤单前挂单深度占比 > 20%
     AND 30min 内重复 ≥ 3 次
  THEN 限制交易 + 审核
```

### 3.2 统计模型

**Z-score 异常检测：**

```
Z = (X - μ) / σ

应用：
- 交易频率 Z-score > 3 → 异常高频
- 单笔金额 Z-score > 4 → 异常大额
- 价格偏离 Z-score > 5 → 异常价格
```

**滑动窗口统计：**
- 短窗口（5min）：检测突发异常
- 中窗口（1h）：检测持续异常
- 长窗口（24h）：检测趋势异常

**四分位距法（IQR）：**
```
IQR = Q3 - Q1
异常下界 = Q1 - 1.5 × IQR
异常上界 = Q3 + 1.5 × IQR
极端异常 = Q3 + 3 × IQR
```

### 3.3 机器学习模型

| 模型 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| Isolation Forest | 无监督异常检测 | 不需要标签、速度快 | 可解释性差 |
| LOF（局部异常因子） | 密度异常 | 适合局部异常 | 计算量大 |
| AutoEncoder | 高维特征 | 自动学习特征 | 需要大量正常数据 |
| XGBoost | 有监督分类 | 准确率高、可解释 | 需要标注数据 |

### 3.4 图算法

**应用场景：**

1. **社区发现（Louvain 算法）**：识别洗钱团伙、对敲群组
2. **PageRank 变体**：评估地址在网络中的重要性/风险传导
3. **最短路径分析**：标记地址到目标地址的资金路径
4. **子图匹配**：识别已知攻击模式（如混币器使用模式）

## 四、实时计算架构

```
Kafka（交易事件流）
  → Flink（窗口聚合 + 特征计算）
    → Redis（实时特征存储）
      → Scoring Service（模型打分）
        → Rule Engine（规则匹配）
          → Action（放行/告警/拦截）
```

**性能要求：**
- 事件延迟：P50 < 10ms, P99 < 50ms
- 吞吐量：> 100,000 events/sec
- 可用性：99.99%

## 五、模型评估

### 5.1 离线评估

| 指标 | 说明 | 计算 |
|------|------|------|
| 精确率 | 告警中真正异常的占比 | TP/(TP+FP) |
| 召回率 | 异常中被检测到的占比 | TP/(TP+FN) |
| F1-Score | 精确率和召回率的调和平均 | 2×P×R/(P+R) |
| AUC | 模型区分能力 | ROC 曲线下面积 |

### 5.2 在线评估

- **告警疲劳度**：日均告警量、误报率趋势
- **规则衰减**：规则命中率随时间的变化（被绕过）
- **覆盖盲区**：通过社区反馈/安全事件复盘发现未覆盖场景

## 六、典型案例

### 6.1 案例：关联账户对敲

**发现过程：**
1. 某交易对成交量突然放大 10 倍，但价格波动 < 0.5%
2. 网络分析发现 30+ 账户在同一 IP 段，使用相同设备指纹
3. 资金链路追踪：所有账户资金最终归集到同一提现地址

**处置：**
- 冻结关联账户群
- 回溯清洗相关交易量
- 将设备和 IP 加入黑名单

### 6.2 案例：API 接口滥用

**发现过程：**
1. 监控系统发现某 API Key 调用频率突增
2. 分析调用模式：每 100ms 下一次单，取消后重新下单
3. 目的：通过高频试探获取做市商报价模型

**处置：**
- API 频率限制调整为 10 req/s
- 该 API Key 降权
- 增加行为序列异常检测规则

## 七、总结

1. **没有银弹**：单一模型无法覆盖所有异常，需要规则+统计+ML+图算法的组合。
2. **特征 > 模型**：好的特征工程比调模型参数更重要。
3. **评估闭环**：必须有回溯测试框架，定期验证模型有效性。
4. **业务理解**：不了解交易业务逻辑，就无法定义什么是"异常"。
5. **成本平衡**：每增加一道检测 = 延迟 + 误拦风险，需要权衡。
