# 研发规范 — 个人要点摘录

> 原规范 14 页，此处只留我写代码时会翻的几条。

## 可迁移性（设计时必想）

服务迁到**另一家交易所**时，缺依赖/表/配置，主流程还能不能跑？  
方案里要有「可迁移性评估」：环境依赖、缺失影响、降级兜底。

## 命名

- 项目名：小写 + 中划线，`risk-` 前缀  
- 反例：驼峰目录、无统一前缀  

## TODO / FIXME

统一格式，带作者，方便搜索。

## 注释

- 复杂业务用注释当「路标」，分段：`// 1、校验 …`  
- **改代码必须改注释**  
- 方法名能说明的，不写冗余 JavaDoc  

## 简单设计四原则（Kent Beck）

1. 通过测试  
2. 少重复  
3. 清晰表达  
4. 更少元素（表达力优先于极简）

## 其他（我记下的）

- 接口、异常、日志、分支规范等见团队完整文档  
- Code Review 关注：安全、性能、可测试性
